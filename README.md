# COPD 照護缺口攔截系統 

> 本專案為個人 Side Project，作為應徵 HTC DeepQ 產品經理職位之作品集提交。
> 以互動式前端原型模擬完整的 AI 輔助 COPD 照護流程，涵蓋院前問診、院內 CDSS 介面與系統架構三個層次。

本作品刻意以 HTC DeepQ 現有的三項核心產品能力為設計骨幹：

| HTC DeepQ 產品能力 | 在本 Demo 中的應用 |
|---|---|
| **LINE AI Chatbot（蘭醫生）** | 院前 COPD-PS 問卷問診、症狀收集、NLP 結構化輸出，患者無需下載額外 App |
| **醫療影像 AI（DeepQ Vision）** | PACS 歷史影像機會性重判、GradCAM 熱力圖標記肺部病徵、Hyperinflation Index 縱向趨勢追蹤 |
| **肺功能分析模組** | Spirometry 多次量測紀錄整合、FEV₁/FVC 趨勢圖、與同齡族群基準比較之異常進展預警 |

---

## 製作動機

COPD（慢性阻塞性肺病）是全球第三大死因，但台灣確診率極低。主因在於：高風險患者大多因其他症狀就診於非胸腔科別，從未觸及肺功能篩檢，形成結構性的「照護缺口」。

本作品試圖回答一個 PM 核心問題：

> **如何將 HTC DeepQ 既有的 AI 產品能力串接成一條完整的照護缺口攔截路徑？**

為此設計了一套三段式攔截策略，並以可互動的前端原型完整呈現其使用者體驗與資料流。

## 功能概覽

### P1 · 院前 LINE 問診

模擬 LINE 官方帳號「蘭醫生呼吸健康管家」的互動流程。

| 功能 | 設計說明 |
|---|---|
| 漸進式問法 | 不直接問「你有 COPD 嗎」，而從爬樓梯、走路距離等生活功能切入，降低病患防衛心理，提高答題真實性 |
| COPD-PS 即時評分 | 五題問卷累積計分，≥ 5 分即自動觸發後端 AI 分析管線 |
| NLP 結構化輸出 | 對話內容萃取後轉為 HL7 FHIR 格式寫入 HIS，醫師不需手動重新輸入病史 |
| 知情同意介面 | 取得病患授權後，方啟動 PACS 歷史影像重判與 Spirometry 紀錄調閱 |

### P2 · 院內 CDSS 介面

模擬醫師實際操作的 HIS 工作站，展示 AI 如何嵌入既有工作流程而不打斷它。

本場景設計三種不同科別的接觸點，說明同一套 AI 核心可橫跨科別主動偵測，無需科別客製化：

| 場景 | 科別 | 切入情境 |
|---|---|---|
| 場景 1 | 胸腔內科 | 主診科別，執行完整 COPD 評估與確診流程 |
| 場景 2 | 家醫科 | 年度健檢掛號，觸發慢性病管理追蹤 |
| 場景 3 | 骨科 | 術前評估時，AI 側向偵測潛在呼吸系統風險 |

**工作站模組**：

| 模組 | 功能說明 |
|---|---|
| 今日候診列表 | AI 高風險病患以黃色標記，醫師開診即可識別，無需主動查詢 |
| 分級警示面板 | 紅（高風險）/ 黃（待確認）兩級，每條警示附信心分數與觸發依據 |
| PACS AI 重判 | 以 GradCAM 熱力圖視覺化標記肺過度充氣、肺氣腫、橫膈扁平等病徵 |
| HIS 病歷時間軸 | 跨年度就診紀錄整合，AI 標記歷次就診中的確診缺口與未跟進項目 |
| EMR 輔助填寫 | AI 根據多來源訊號自動產生 SOAP 診斷摘要草稿，醫師審閱後確認 |
| ICD-10 建議 | 主診斷、疑似診斷、繼發診斷碼自動建議，並標示與現有診斷碼的差異 |
| 轉診單產生器 | 一鍵產生結構化 SOAP 格式轉診文件 |
| Human-in-the-Loop | 所有 AI 建議皆需醫師明確確認後方執行，符合 TFDA SaMD 輔助決策分類規範 |

## 系統設計重點

### 為什麼選擇 LINE 而非 App？

台灣 LINE 滲透率逾 90%，不需要病患下載額外 App，大幅降低院前接觸的摩擦成本。此外 LINE Official Account 支援 Webhook 與結構化訊息，可直接串接後端 NLP 管線。

### 為什麼做機會性影像探勘（Opportunistic Screening）？

要求病患新照一張 X 光作為篩檢依據，在臨床流程中幾乎不可行。本設計改為重判病患**已存在於 PACS 中的歷史影像**，零成本取得額外資訊，不需動用醫師或放射師時間。

### Human-in-the-Loop 不是妥協，而是設計選擇

TFDA 現行對 AI 輔助決策軟體（SaMD）的要求，以及醫師對 AI 誤判的責任歸屬疑慮，使得「AI 直接執行」在台灣臨床場域目前不可行。本設計將 AI 定位為「加速醫師判斷」而非「取代醫師判斷」，是在法規現實下最可落地的策略。

## 技術規格

| 項目 | 說明 |
|---|---|
| 形式 | 前端互動原型（UI Prototype） |
| 實作 | 純 HTML / CSS / JavaScript，單一檔案，無需建置環境 |
| 介面語言 | 繁體中文（台灣） |
| 系統整合（設計層） | HL7 FHIR R4、REST API、LINE Messaging API |
| 法規框架（設計層） | TFDA SaMD 輔助決策分類、個人資料保護法 |
| 架構模式 | Human-in-the-Loop |
| 瀏覽器支援 | Chrome / Edge / Safari，建議桌機寬螢幕檢視 |

> **開發方式**：本專案以 [Claude Code](https://claude.ai/code) 作為主要開發工具，由本人負責產品定義、使用者流程設計與所有技術規格的指定，並透過 AI 協作完成實作。這也是本作品希望體現的核心能力之一：PM 以 AI 工具獨立交付可互動原型，縮短從概念到驗證的週期。

本 Demo 為純前端靜態頁面，無需後端伺服器或任何安裝步驟：

## Demo 操作建議路徑

**P1 院前問診**：點擊畫面左側的選項按鈕，依序完成問卷，觀察 COPD-PS 分數累積至 ≥ 5 分後的 AI 觸發流程與知情同意介面。

**P2 院內 CDSS**：
1. 點擊候診列表中橘黃色高亮的病患列，進入醫師工作站
2. 切換上方「場景 1 / 場景 2 / 場景 3」，觀察同一 AI 核心在不同科別的介面呈現差異
3. 點擊「查看影像」進入 PACS AI 重判頁，觀察 GradCAM 熱力圖標記區域

**P3 系統架構**：點擊 SVG 圖中任一模組節點，右側面板顯示對應的技術規格與設計決策說明。
